Ploinky Wormhole Documentation

This project provides the public rendezvous and signaling server for did:wormhole. The server stores minimal public DID state, transports temporary communication intents and intent responses, and relays temporary signaling data used to establish WebRTC or compatible P2P channels.

Trust semantics, policy enforcement, contact classification, and final content transfer remain outside the server and are handled by Wormhole agents. The server implements operational safeguards only: TTL expiration, payload limits, rate limits, and optional blocklists.

Documentation and specifications

The DS specification set is authoritative. Read the matrix through specsLoader matrix entry, then follow the relevant DS files for implementation constraints.

Coding style, module boundaries, and test organization are defined in DS001-coding-style.md.

Runtime defaults

Runtime is Node.js ESM with no external dependencies. DID state persists in a local JSON file, while intents, responses, signaling queues, replay cache, and relay cache are in-memory temporary state with TTL cleanup.

Testing organization

Tests use built-in node:test and execute request-level behavior against an ephemeral server instance. Coverage focuses on DID state publishing, DID-protected reads, signaling transport behavior, and anti-abuse constraints.
