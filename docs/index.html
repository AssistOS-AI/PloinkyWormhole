Ploinky Wormhole Documentation

This project provides the public rendezvous and signaling server for did:wormhole. The server stores minimal public DID state, transports temporary intents and intent responses, and relays temporary signaling data used to establish WebRTC or compatible P2P channels.

The protocol surface remains semantically neutral. Trust semantics, workspace policy, contact classification, and final content transfer remain outside the server and are handled by Wormhole agents. Operationally, the runtime now also exposes readiness checks, admin observability, persistent manual blocklists, bounded automated abuse controls, and a token-driven dashboard shell.

Production note: read the new Operations and Deployment pages together with DS005 and DS006 before exposing the service publicly.

Runtime defaults

Runtime is Node.js ESM with no external dependencies. DID state and manual admin blocklists persist in local JSON files. Intents, responses, signals, replay / relay caches, observability event buffers, and auto-block state remain bounded in memory with periodic cleanup.

Testing organization

Tests use built-in node:test and execute request-level behavior against ephemeral server instances. Coverage includes DID state publication, DID-protected reads, cross-server forwarding, admin surfaces, trusted-proxy behavior, and abuse controls.
