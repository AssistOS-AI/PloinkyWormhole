Security and Anti-Abuse Controls

The server enforces operational protections without semantic trust classification. It does not decide whether a sender is trusted or known; it limits abuse, preserves bounded resources, and keeps sensitive metadata queues protected by DID or admin authentication.

Boundary controls

Requests are constrained by maximum body size, envelope size, signal payload size, maximum signals per intent, replay-cache caps, relay-cache caps, and bounded observability buffers. These limits make the runtime predictable under load.

Rate limiting and auto-blocks

Per-IP limiting applies to every request. DID-keyed limits apply to DID-to-DID writes and are enforced after successful signature verification for signed flows. Repeated abuse raises offense scores and can trigger temporary IP or verified-DID auto-blocks.

Unsigned intent submission never auto-blocks claimed DIDs. This prevents attackers from forging identities and weaponizing the abuse pipeline.

Trusted proxy model

The runtime trusts x-forwarded-for only when trusted-proxy mode is explicitly enabled and the immediate upstream address is loopback or listed in TRUSTED_PROXY_IPS. Direct-exposure deployments should keep trusted-proxy mode disabled.

Security headers and admin token handling

Responses set X-Content-Type-Options, X-Frame-Options, Referrer-Policy, restrictive CSP, and Cache-Control: no-store. Admin APIs require x-admin-token; token transport in URLs is intentionally unsupported.

Timeout hardening

Request timeout, header timeout, socket timeout, keep-alive timeout, and max requests per socket reduce the impact of slowloris-style and connection-hoarding attacks. Reverse proxies should mirror or tighten these limits at the edge.

Scope boundary

The server is still intentionally neutral: it does not host mailboxes, chat streams, file transfer, task payloads, or workspace policy engines. Hardening and observability improve operational safety without changing protocol semantics.
