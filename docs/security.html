Security and Anti-Abuse Controls

The server enforces operational protections without semantic trust classification. It does not decide if a sender is trusted, known, or blocked at application semantics level. It only applies transport-level controls.

Operational controls

Controls include rate limits by IP and DID dimensions, payload size limits, per-intent signaling queue limits, short TTL windows for temporary objects, replay cache for signed requests, and relay cache for forwarded object deduplication.

Signatures use the current DID public key and canonical request payloads. DID-protected reads are mandatory for intents, responses, and signaling retrieval so metadata remains private to DID controllers.

Constraints and scope

The server is intentionally neutral: it does not host mailboxes, chat streams, file transfer, task payloads, or workspace policy engines. Final content is exchanged peer-to-peer after agent-level acceptance and channel establishment.
