API Surface

The API mirrors three server concepts: DID state, communication intent, and signaling.

MethodEndpointPurposeSignature required
GET/did-state?did=...Read public DID stateNo
PUT/did-stateCreate or update DID stateYes after bootstrap
POST/intentSubmit intent or receive forwarded intentNo
GET/intents?to_did=...Read active intents for DIDYes (to_did)
POST/intent/{intent_id}/seenMark intent as seenYes (to_did)
POST/intent-responseSubmit response to intentYes (from_did responder)
GET/intent-responses?to_did=...Read responses addressed to DIDYes (to_did)
POST/signalSubmit signaling messageYes (from_did)
GET/signals?to_did=...&intent_id=...Read signaling messagesYes (to_did)
GET/healthLiveness statusNo
GET/limitsPublished server limitsNo

Forwarding semantics

The same endpoints process local and forwarded objects. If to_did belongs to another server domain, the object is forwarded over HTTPS according to server configuration. Relay metadata headers prevent loops and duplicate delivery. For signed transport objects (/intent-response and /signal), forwarding preserves the original DID signature headers and signed body so the destination server can re-verify canonical signatures, including remote DID-state resolution when sender DID state is not local.

Production cross-server forwarding is HTTPS-only. Insecure protocol use is blocked unless explicitly enabled for loopback-scoped local testing.

DID-protected endpoints require an active DID state for authentication. DIDs marked disabled or revoked remain publicly visible through /did-state but cannot authorize protected metadata operations.
