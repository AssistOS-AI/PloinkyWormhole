Deployment and Hardening

The runtime is designed to sit behind a reverse proxy or edge load balancer. Safe production use depends on reverse-proxy hygiene, strong bootstrap and admin secrets, durable storage for DID and admin state, and keeping the runtime's timeout and abuse controls enabled.

Recommended topology

Terminate TLS at the edge, bind the Wormhole runtime to loopback or a private address, enable trusted-proxy mode only when the upstream proxy is trusted, and explicitly set TRUSTED_PROXY_IPS for non-loopback proxies.

Mandatory production settings

SettingExpectation
BOOTSTRAP_TOKENNon-default, 32+ random characters.
ADMIN_TOKENConfigured, 32+ random characters.
FORWARD_PROTOCOLhttps.
ALLOW_INSECURE_FORWARDINGfalse.
DID_STATE_FILE, ADMIN_STATE_FILEDurable filesystem paths with backups.
Timeout settingsKeep request, header, socket, and keep-alive timeouts enabled.

Reverse-proxy responsibilities

The proxy should rewrite x-forwarded-for, enforce body limits, enforce edge timeouts, add HSTS, and optionally apply coarse IP rate limiting before traffic reaches the runtime. Admin surfaces should be reachable only from trusted operator paths.

Readiness workflow

Treat /ready as the deployment gate. A healthy but degraded process is still a rollout blocker when tokens are weak, forwarding is unsafe, or trusted-proxy mode is misconfigured.
