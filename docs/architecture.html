Architecture

The server is a single-process HTTP runtime with explicit boundaries between durable DID/operator state and bounded temporary transport state. Recipient-side local intents remain the canonical lifecycle records used to validate local responses and accepted signaling.

      A[Agent Sender] -->|POST /intent| B[Server Domain A]
      B -->|HTTPS forward| C[Server Domain B]
      C -->|GET /intents signed by to_did| D[Agent Recipient]
      D -->|POST /intent-response signed| C
      C -->|HTTPS forward| B
      E[Operator] -->|x-admin-token| F[/admin/metrics /events /blocklist]
      F --> G[observability + admin-state]

Core modules

src/server.mjs starts the HTTP listener and applies request / connection timeouts. src/app.mjs handles routing, validation, signatures, forwarding, admin APIs, and queue management. src/storage.mjs persists DID states. src/admin-state.mjs persists manual blocklists. src/observability.mjs tracks request metrics, recent security events, top offenders, attack level, and auto-block TTL state. src/signatures.mjs, src/rate-limit.mjs, src/did.mjs, and src/config.mjs keep crypto, throttling, DID parsing, and config logic isolated.

State partitions

Durable state persists in local JSON files with atomic writes: DID state and manual admin blocklists. Temporary state lives in memory as bounded maps and buffers: visible intents, canonical local intent states, responses, signals, replay cache, relay cache, recent security events, offender scores, and active auto-blocks.

Canonical signing model

DID-protected operations require x-did, x-signature, x-timestamp, and optional x-nonce. The server verifies the canonical payload below against the current DID public key: 

METHOD + "\n" + PATH_WITH_QUERY + "\n" + TIMESTAMP + "\n" + NONCE + "\n" + SHA256(BODY_RAW)

Requests outside the clock-skew window are rejected. Replays are blocked through a bounded replay cache. For cross-server forwarded signed requests, the destination resolves DID state remotely when needed.

Trusted proxy model

Source IP comes from socket.remoteAddress by default. x-forwarded-for is only honored when trusted-proxy mode is enabled and the immediate upstream address is loopback or explicitly trusted. This keeps reverse-proxy deployments compatible without allowing direct client spoofing.

Lifecycle enforcement

Visible intents represent only work still awaiting recipient action. Final responses remove the intent from the visible queue immediately. Signals are accepted only when the server can prove a local canonical intent is already accepted; otherwise they are rejected or delegated to the peer server that owns that canonical record.
