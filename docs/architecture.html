Architecture

The server is a single-process HTTP runtime with explicit boundaries between durable DID state and temporary transport queues. Durable state is limited to DID documents and key history. Temporary state contains intents, responses, signaling messages, and replay/relay caches. 

flowchart LR
      A[Agent Sender] -->|POST /intent| B[Server Domain A]
      B -->|HTTPS forward| C[Server Domain B]
      C -->|GET /intents signed by to_did| D[Agent Recipient]
      D -->|POST /intent-response signed| C
      C -->|HTTPS forward| B
      D -->|POST /signal signed| C
      C -->|GET /signals signed| D

Core modules

src/server.mjs starts the HTTP listener and wires lifecycle. src/app.mjs handles routing, validation, signatures, forwarding, and queue management. src/storage.mjs persists DID states through atomic JSON writes. src/signatures.mjs defines canonical signature payloads and verification. src/rate-limit.mjs enforces fixed-window throttling with backoff.

Canonical signing model

DID-protected operations require x-did, x-signature, x-timestamp, and optional x-nonce. The server verifies signatures against the current DID public key over: 

METHOD + "\n" + PATH_WITH_QUERY + "\n" + TIMESTAMP + "\n" + NONCE + "\n" + SHA256(BODY_RAW)

Requests outside allowed clock skew are rejected. Replayed signatures are blocked within a replay window.
